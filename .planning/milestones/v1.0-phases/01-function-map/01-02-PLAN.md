Implementation in fmap.cjs -- add `const { execSync } = require('child_process');` at the top imports:

```javascript
/**
 * Detect files changed since last commit (both staged and unstaged).
 * Used by gsd-cataloger for incremental updates (FMAP-05, D-04).
 * Returns JSON array of changed file paths (POSIX-normalized, relative to project root).
 */
function cmdFmapChangedFiles(cwd, args, raw) {
  const codeExtensions = ['.ts', '.js', '.cjs', '.mjs', '.tsx', '.jsx', '.vue', '.php', '.py', '.rb', '.go', '.rs', '.java'];

  let files = new Set();

  try {
    // Unstaged + staged changes vs HEAD
    const diffHead = execSync('git diff --name-only HEAD 2>/dev/null || true', { cwd, encoding: 'utf-8' }).trim();
    if (diffHead) diffHead.split('\n').forEach(f => files.add(f));
  } catch { /* no git or no HEAD */ }

  try {
    // Staged changes (for files added but not yet committed)
    const diffCached = execSync('git diff --name-only --cached 2>/dev/null || true', { cwd, encoding: 'utf-8' }).trim();
    if (diffCached) diffCached.split('\n').forEach(f => files.add(f));
  } catch { /* no git */ }

  try {
    // Untracked new files
    const untracked = execSync('git ls-files --others --exclude-standard 2>/dev/null || true', { cwd, encoding: 'utf-8' }).trim();
    if (untracked) untracked.split('\n').forEach(f => files.add(f));
  } catch { /* no git */ }

  // If --since-commit <hash> is passed, use diff from that commit to HEAD instead
  const sinceIdx = args ? args.indexOf('--since-commit') : -1;
  if (sinceIdx !== -1 && args[sinceIdx + 1]) {
    files = new Set();
    try {
      const diffSince = execSync(`git diff --name-only ${args[sinceIdx + 1]} HEAD 2>/dev/null || true`, { cwd, encoding: 'utf-8' }).trim();
      if (diffSince) diffSince.split('\n').forEach(f => files.add(f));
    } catch { /* bad commit ref */ }
  }

  // Filter to code files only
  const codeFiles = [...files]
    .filter(f => codeExtensions.some(ext => f.endsWith(ext)))
    .map(f => toPosixPath(f))
    .sort();

  output({ files: codeFiles, count: codeFiles.length }, raw);
}
```

**`<role>` section:**
"You are the GSD cataloger agent. You populate and maintain `.planning/function-map.json` -- a flat JSON registry of all significant functions, methods, and classes in the project. You are a CHEAP MODEL (haiku). Be efficient: minimize token usage, avoid reading entire files when tool output suffices."

**Provider Detection section:**
Step 1: Probe Serena availability by calling `mcp__serena__get_symbols_overview` on a known small file (first `.ts` or `.js` file found). If it succeeds, use Serena path. If it fails or returns empty, switch to grep path immediately.

**Serena Path section** (implements FMAP-03, per D-01, D-02):
For each target file:
1. Call `mcp__serena__get_symbols_overview({ relative_path: "<file>" })` to get top-level symbols (classes, exported functions, exported constants).
2. For each class found: call `mcp__serena__find_referencing_symbols({ symbol_name: "<ClassName>", relative_path: "<file>" })` to check external callers.
3. For each method of the class: call `mcp__serena__find_referencing_symbols` for the method name. If ANY caller is outside the current file, include the method as a map entry (per D-01: methods only when they have external callers). If all callers are internal, skip.
4. Build JSON patch with entries keyed as `<relative_posix_path>::<ClassName>::<methodName>` (or `<relative_posix_path>::<functionName>` for top-level functions).
5. For each file processed, use `--replace-file` flag: run `node gsd-tools.cjs fmap update --replace-file <filepath> --data '<json>'` to remove stale entries before adding new ones (handles deleted functions per Pitfall 5 from RESEARCH.md).

**Grep Fallback section** (implements FMAP-04, per D-11, D-12, D-13):
When Serena is unavailable:
1. Run grep with `--include` flags for code extensions to find candidate symbols: `grep -rn --include="*.ts" --include="*.js" --include="*.cjs" --include="*.mjs" --include="*.vue" --include="*.php" -E "(export (async )?function|export (default )?class|export const .+= (async )?\(|module\.exports)" <target_dirs>`
2. Read the grep output. For each candidate: extract function/class name, signature (from the line + next few lines if needed), kind (function/method/class/arrow), exported status, language (from file extension).
3. For callers: run `grep -rn "<functionName>" --include="*.ts" --include="*.js" --include="*.cjs" <project_dirs>` and interpret results to build callers[] array. Exclude the definition site itself.
4. Build JSON patch with IDENTICAL schema to Serena path (per D-13).
5. Update map using same `--replace-file` pattern.

**Entry Schema section** (per D-08):
Document that every entry MUST have: kind (function/method/class/arrow), signature, purpose, callers[] (file:line format), calls[] (file::function format), language (ts/js/cjs/vue/php), exported (boolean), last_updated (ISO timestamp). Show the full JSON example from RESEARCH.md.

**Key Format section:**
- Top-level functions: `relative/path/to/file.ts::functionName`
- Class methods: `relative/path/to/file.ts::ClassName::methodName`
- Always use POSIX slashes, never leading `./`

**Incremental vs Full Scan section** (implements FMAP-05, per D-04, D-05):
- Default mode (incremental): Run `node gsd-tools.cjs fmap changed-files` to get the list of changed code files. Only process those files.
- Full scan mode (when invoked with `--full` argument): Process all code files in the project. Used as safety net or on first run.
- On first run (when `node gsd-tools.cjs fmap stats` shows total: 0): automatically do a full scan.

**Concurrency Warning** (per Pitfall 4 from RESEARCH.md):
NEVER run fmap updates inside parallel wave tasks. The cataloger should run BEFORE or AFTER a wave, never during. If spawned by execute-phase, it runs as a pre-execution sequential step.

**Anti-Patterns section:**
- NEVER read function-map.json directly -- always use `node gsd-tools.cjs fmap get` or `fmap update`
- NEVER store return_type or param_types in the map (per D-09)
- NEVER use complex regexes for grep fallback -- you (the LLM) interpret the raw grep output
- NEVER write the full map on every update -- use `--replace-file` per-file updates
